# M3/M4 Canonical Disposition

This document records the M3A boundary characterization for the approved
M3/M4 refactor. It does not move runtime code, alter artifact schemas, or
change command inventory.

## Canonical Path B

The canonical native Path B is the exact configuration combining
`corridor_exemplar_v1`, `corridor_first_global_backfill_v1`, enabled exemplar
selection, and `two_pass_rerun_selected`. Its preserved order is:

`preflight -> score pass -> early corridor materialization -> selection authority export -> C2-C5 selection -> selected rerun -> late corridor finalization -> assembly -> validation/linkage -> reconciliation/cover -> reporting`.

Early and late corridor materialization intentionally use the same deterministic
builder but have different authority:

| Phase | Current caller | Public-artifact state | Progress and resume meaning |
|---|---|---|---|
| Early score-surface corridor materialization | `builder.production.build_production_gpu_tome` | Writes provisional `corridors/*` from the completed score store with zero selected records and no selected-link claim. | It is a lightweight `fingerprint_corridor_export` checkpoint with no delivery callback. Its files support selection authority export only. |
| Late selected-artifact corridor finalization | `builder.exemplar_delivery.materialize_selected_exemplar_delivery` | Overwrites the public `corridors/*` after selected payload summaries exist, attaches selected records to fingerprint/mode IDs, and is the final public corridor state. | Emits detailed `corridor_export` events through `assignments_written`; only this final selected-linked surface is delivery/validation/report evidence. |

M3A characterization verifies the early call happens before selected delivery,
the selected rerun completes before the late call, and detailed corridor-export
progress is emitted only by the late call. The final public corridor summary is
the late summary: it has the selected count and selected-linkage truth, while
the early snapshot has count zero and linkage false.

Finalization-only resume remains evidence-derived. With complete score,
selection, selected-delivery, and final corridor evidence, removing only the
C6 integrated-selection validation report leaves the existing probe eligible
for CPU-only finalization. This is characterization of the current seam, not a
new persistent stage schema or a broader resume promise.

## M4B/M4C Native Boundaries

For the exact canonical request, the production facade now executes preflight
and score pass through typed slice-one callbacks before continuing into the
preserved corridor/selection/delivery implementation. Global-only requests do
not enter that adapter. The callback seam preserves the existing report,
progress, and runtime-error behavior; it adds no artifact or progress schema.

`native_path_b.resume` is a separate, read-only M4C classifier. It derives the
earliest action from the existing run plan, streaming manifest, early or late
corridor readers, C6 authority/selection evidence, delivery report, validation,
reconciliation, and terminal report. A provisional early corridor can only
advance to authority/selection or selected delivery; it is never final evidence.
After late finalization overwrites the public corridor surface, corruption of
that surface belongs to late-corridor repair. Optional full production-config
context verifies the persisted selection-integration hash, while the existing
compatibility migration remains the production owner's responsibility.

## Preserved and Research Disposition

Canonical runtime code remains the production facade, streaming score pass,
corridor artifacts, C6 selection integration, selected delivery, strict
selected-linkage audit, target/provenance/I/O contracts, run plan/doctor, and
cover/package helpers. `builder.teacher_textbook` remains a leaf compatibility
dependency during M3; M4 may extract only proven required types or validators.

Research-frozen surfaces remain explicitly invocable and preserved: experimental
multi-GPU Path B; HF specimen/export/Qwen-policy experiments; generalized
fingerprint artifact, generation, measurement, inspection, and summary
surfaces; historical capability proofs; and older prompt/split/tokenization
routes. They are not deleted, renamed, or silently routed through canonical
Path B in M3A.

Compatibility remains a contract: existing public names and parser inventory
stay available during M3/M4, legacy artifacts remain inspectable/finalizable
through existing readers and migrations, and no algorithm, artifact path, or
fixture is regenerated by this checkpoint. Future M3B work may isolate imports
through lazy compatibility exports; M3A establishes behavior before that move.
